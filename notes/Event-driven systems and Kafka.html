Event-driven systems and Kafka

Why event-driven systems

When you start a company with 2-3 engineers usually you don't need event driven systems. It makes sense to use an async architecture if you have a big apllication and several teams working on it. Linkedin created Kafka to cope with their growth.

Commands, events, and queries

Command is an action that must be performed by remote service in a synchronous manner.

Event is a business fact that something happened. Usually it makes sense to persist a business fact. Producer sends an event and forgets about it.

Query is a request to get data. Query doesn't change state.

Kafka

We can use Kafka streams to query necessary data. Several streams can be combined to create a new stream.

Kafka stores data locally to avoid network calls. Kafka is distributed so it will continue working if a partition fails.
